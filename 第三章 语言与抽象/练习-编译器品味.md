**(a) 品味的具体表现**

**(a) Concrete manifestations of taste**

本章指出："编译器将编程语言设计者的品味施加给了所有的普通程序员"。选两种你熟悉的编程语言（如 C 与 Python，或 Go 与 Rust，或 Java 与 JavaScript），对比它们的设计者通过编译器/解释器"强制"了哪些不同的品味。每种语言至少列出两点。

The chapter notes: "the compiler imposes the programming language designer's taste upon all ordinary programmers." Choose two programming languages you are familiar with (e.g., C vs. Python, Go vs. Rust, Java vs. JavaScript). Compare what different "tastes" their designers enforce through the compiler/interpreter. List at least two points for each language.

*示例方向（不要照抄）：语法风格（缩进强制 vs 大括号自由）、类型检查的严格程度、错误处理的哲学（异常 vs 返回值）、内存管理的显式程度。*

*Example directions (do not copy): syntax style (forced indentation vs. free braces), strictness of type checking, error handling philosophy (exceptions vs. return values), explicitness of memory management.*

**(b) 品味如何影响代码质量**

**(b) How taste affects code quality**

选 (a) 中一种语言的某个"品味约束"，分析它如何在实际工程中影响代码质量。考虑：这个约束是否减少了某种常见的 bug？它是否降低了代码的可读性或编写速度？综合来看，这个约束对"软件质量"的净效果是正还是负？

Choose one "taste constraint" from one of the languages in (a) and analyze how it affects code quality in practice. Consider: does this constraint reduce a common type of bug? Does it reduce code readability or writing speed? On balance, is the net effect of this constraint on "software quality" positive or negative?

**(c) What if the LLM becomes the compiler**

In the framework at the end of this chapter, the rules of a "Restricted Natural Language" serve as the compiler of the LLM era — imposing the CNL designer's taste on those who program with AI. Based on the CNL rules you designed in Exercise 1, answer: Do your CNL rules contain some kind of "taste judgment"? What new ones might it introduce?; If a programmer finds your taste "wrong," what options do they have?
