# 练习 3 — 哈希表：为什么开放寻址在现代 CPU 上更快

Exercise 3 — Hash Tables: Why Open Addressing is Faster on Modern CPUs

传统教材教两种哈希表冲突解决策略：拉链法（每个桶是一个链表）和开放寻址法（冲突时探测下一个空位）。在渐近复杂度上，两者都是 O(1) 期望时间。但在实际性能上，开放寻址通常大幅领先——因为现代 CPU 的缓存层级。

Traditional textbooks teach two hash table collision resolution strategies: chaining (each bucket is a linked list) and open addressing (upon collision, probe the next empty slot). Asymptotically, both are O(1) expected time. In practice, open addressing is usually much faster — because of modern CPU cache hierarchies.

---

**(a) 模拟两种策略的缓存行为**

考虑一个包含 500 个元素的哈希表，拉链法使用链表节点（每个节点 24 字节，分散在堆上），开放寻址法使用单个数组（连续内存）。当处理器需要访问一个元素时，拉链法需要先访问桶指针（在数组内），然后追踪链表节点（在堆上随机位置）。开放寻址法只需要在数组内线性扫描。对于这两种策略，一次成功查找平均各需要多少次**缓存行加载**（假设缓存行 = 64 字节）？

**(a) Simulate cache behavior of both strategies**

Consider a hash table with 500 elements. Chaining uses linked list nodes (24 bytes each, scattered on the heap). Open addressing uses a single array (contiguous memory). When the processor accesses an element, chaining first accesses the bucket pointer (in the array), then follows the linked list node (at a random heap location). Open addressing only scans linearly within the array. How many **cache line loads** does a successful lookup require on average for each strategy (assume cache line = 64 bytes)?

*提示：拉链法的链表节点在堆上分配，位置不可控——几乎每个节点都需要一次全新的缓存行加载。开放寻址的数组是连续的：一次探测可能跨越多条缓存行，但一旦探测偏移落在一个缓存行内，后续位置已经在该缓存行中。Hint: Chaining's nodes are heap-allocated at uncontrollable locations — nearly every node requires a fresh cache line load. Open addressing's array is contiguous: probes may span multiple cache lines, but once a probe offset lands within a cache line, subsequent positions are already in that cache line.*

---

**(b) SIMD 与"批量探测"**

现代高性能哈希表（如 SwissTable / Abseil flat_hash_map）用 **SIMD 指令**一次并行探测 16 个桶。每个桶存储 1 字节的元数据（哈希值的高 7 位），使得 16 个桶的元数据恰好占据 16 字节——一个 SSE 寄存器。一次 SIMD 比较就可以找出 16 个桶中哪些可能匹配。一个 16 个桶的"邻域"占多少字节？它在 L1 缓存（32 KB）中占据多少比例？这如何体现了"层级意识"——将数据结构的大小调整到恰好匹配某一层缓存的容量？

**(b) SIMD and "bulk probing"**

Modern high-performance hash tables (e.g., SwissTable / Abseil flat_hash_map) use **SIMD instructions** to probe 16 buckets in parallel. Each bucket stores 1 byte of metadata (the high 7 bits of the hash), so 16 buckets' metadata occupies exactly 16 bytes — one SSE register. A single SIMD comparison identifies which of the 16 buckets might match. How many bytes does a "neighborhood" of 16 buckets occupy? What proportion of L1 cache (32 KB) does it represent? How does this reflect "hierarchy awareness" — sizing the data structure to exactly match the capacity of one level of cache?

---

**(c) "哈希表是 O(1) 因为局部性"**

哈希表是渐近 O(1) 的——这是信息论的结果（第二章：均匀哈希下的熵计算）。但它**常数极佳**的原因则是局部性：开放寻址 + SIMD 批量探测将探测序列压缩进连续的内存块，使得大部分查找命中 L1 或 L2 缓存。用本章的"层级表"解释：为什么一个 L1 缓存命中（~1 ns）的哈希表查找，比一个需要访问 DRAM（~100 ns）的查找，快了约两个数量级？

**(c) "Hash tables are O(1) because of locality"**

Hash tables are asymptotically O(1) — this is an information-theoretic result (Chapter 2: entropy calculations under uniform hashing). But the reason their **constant factors are excellent** is locality: open addressing + SIMD bulk probing compresses the probe sequence into contiguous memory blocks, so that most lookups hit L1 or L2 cache. Using this chapter's hierarchy table, explain: why is a hash table lookup that hits L1 cache (~1 ns) roughly two orders of magnitude faster than one that must access DRAM (~100 ns)?
