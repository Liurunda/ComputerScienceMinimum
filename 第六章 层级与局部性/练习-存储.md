# 练习 5 — 存储：层级与局部性原则在硬件和文件系统中的应用

Exercise 5 — Storage: Applying Hierarchy and Locality Principles to Hardware and Filesystems

本章的核心命题——局部性使得层级成立——不是 CPU 缓存的专利。存储硬件和文件系统的整个历史，就是不断在不同尺度上重新实现同一套原则。

This chapter's core proposition — locality makes hierarchy possible — is not the exclusive domain of CPU caches. The entire history of storage hardware and filesystems is the repeated re-implementation of the same set of principles at different scales.

**(2)** RAID 是"用冗余换可靠性"的存储层级策略。RAID 1（镜像）将同一份数据写入两个盘——两倍空间，容忍一个盘故障。RAID 5 将数据条带化并写一个校验块——N+1 个盘，容忍一个盘故障，空间开销 1/N。这两种策略在"空间开销""读写性能""故障容忍度"三个维度上做了不同的 tradeoff。将这三种维度映射到第六章的"层级表"中的"容量、延迟、层级数"——它们是否同构？

RAID is a storage hierarchy strategy of "trading redundancy for reliability." RAID 1 (mirroring) writes the same data to two disks — 2× space, tolerates one disk failure. RAID 5 stripes data and writes one parity block — N+1 disks, tolerates one disk failure, space overhead 1/N. These two strategies make different tradeoffs along three dimensions: "space overhead," "read/write performance," and "fault tolerance." Map these three dimensions onto the "capacity, latency, number of levels" from this chapter's hierarchy table — are they isomorphic?

**(3)** SSD 内部有一个"闪存翻译层"（FTL），它维护了一张从"逻辑块地址"到"物理闪存页"的映射表。当操作系统要求"覆写第 100 块"时，FTL 并不真的覆写原位置——它将新数据写入一个全新的、已经擦除过的物理页，然后更新映射表，将旧页面标记为垃圾。这个策略被称为"异地更新"（out-of-place update）。为什么 SSD 要做这件事？这和你已经了解的"虚拟内存把逻辑地址映射到物理地址"在结构上有什么相似性和差异？

Inside an SSD, a Flash Translation Layer (FTL) maintains a mapping table from "logical block address" to "physical flash page." When the OS requests "overwrite block 100," the FTL does not actually overwrite the original location — it writes new data to a brand-new, already-erased physical page, updates the mapping table, and marks the old page as garbage. This strategy is called "out-of-place update." Why does an SSD do this? How is it structurally similar to and different from what you already know about "virtual memory mapping logical addresses to physical addresses"?

*提示：闪存的"擦除前不能覆写"特性。Hint: flash memory's "must erase before overwrite" constraint.*

**(4)** 数据库通常不直接把表存储在文件系统上——它们在文件系统之上再实现一层自己的存储引擎（如 B 树、LSM 树），维护自己的页缓存，并绕开操作系统缓存。为什么一个数据库需要"双重层级"——操作系统的文件缓存已经有一层了，为什么数据库还要在它之上再做一层？

Databases typically do not store tables directly on the filesystem — they implement their own storage engine on top (e.g., B-trees, LSM trees), maintain their own page cache, and bypass the OS cache. Why does a database need a "double hierarchy" — the OS already has a file cache; why build another layer on top?

*提示：数据库知道哪些页"在同一个事务内被一起访问"。OS 不知道。Hint: the database knows which pages are "accessed together within the same transaction." The OS does not.*
