# 练习 5 — 在随机性中维持确定性：不变量与正确性证明

Exercise 5 — Maintaining Determinism Within Randomness: Invariants and Correctness Proofs

第四章的核心张力是：你接受一点"可能出错"的概率，换来效率。但有些东西不能"概率上正确"——飞行控制软件、心脏起搏器、加密货币合约必须被证明在所有可能的执行路径上都满足规约。如何在随机的世界中证明确定性？

The central tension of Chapter 4: you accept a small probability of being wrong in exchange for efficiency. But some things cannot be "probabilistically correct" — flight control software, pacemakers, cryptocurrency contracts must be proven to satisfy their specification on all possible execution paths. How do you prove determinism in a random world?

---

**(1)** 不变量是一个在程序执行的每一条路径——无论if走哪边，无论循环执行几次——上都保持为真的命题。一个不变量不依赖"程序这次运气好不好"。在一个随机化快速排序中，有哪些不变量在每一次随机选择pivot后依然成立？

An invariant is a proposition that remains true on every execution path — regardless of which branch of an if is taken, regardless of how many times a loop iterates. An invariant does not depend on "whether the program is lucky this time." In a randomized quicksort, what invariants still hold after each random pivot selection?

*提示："pivot已经在最终排序位置上"。Hint: "the pivot is already in its final sorted position."*

---

**(2)** 循环不变量是证明循环正确性的最基本形式化方法。给定以下伪代码：

```text
// 在数组 A[0..n-1] 中查找值 x
i := 0
while i < n:
    if A[i] == x:
        return i
    i := i + 1
return -1
```

写出一句循环不变量——一个在每次循环体执行前一定为真的命题——并用它来证明：如果函数返回 −1，那么 x 确实不在 A 中。

A loop invariant is the most basic formal method for proving loop correctness. Given the pseudocode above, write a loop invariant — a proposition that must be true before each execution of the loop body — and use it to prove: if the function returns −1, then x is indeed not in A.

---

**(3)** 类型系统是一种"自动化的轻量级形式化证明"。当编译器接受 `int x = "hello"` 为类型错误时，它已经自动证明了这个程序在类型层面上不会犯某一类错误。将类型系统视为一种"静态不变量"——在"执行前"就已确立的正确性保证——它和 fuzzing（练习 4 中你实现的随机测试工具）在哲学上的根本分歧是什么？哪一种能给你"这个程序不会把 int 赋给 string"的绝对保证？

A type system is a kind of "automated lightweight formal proof." When the compiler rejects `int x = "hello"` as a type error, it has already automatically proven that this program will not commit a certain class of errors at the type level. Treating the type system as a "static invariant" — a correctness guarantee established "before execution" — what is the philosophical divergence between type-checking and fuzzing (the random testing tool you implemented in Exercise 4)? Which one can give you the absolute guarantee that "this program will never assign a string to an int"?
